# 风控技能编写模板（Risk Skill Authoring Template）

> 参考来源：阿里云点金 DianJin-SKILLS（百技图）credit-risk-manager 角色技能的工程模式，
> 重点精读了 `credit-risk-cot`（风险逆向思维链）与 `loan-risk-monitor`（贷后五维监测）。
> 仓库：https://github.com/aliyun/qwen-dianjin/tree/master/DianJin-SKILLS/credit-risk-manager
> 注意：该仓库无 LICENSE，可参考其工程模式自行撰写，不要整段复制原文进对外交付物。
>
> 用法：新建风控类技能时，复制本模板为 `SKILL.md`，按章节逐项填写。
> 标注【必填】的章节不允许省略——风控技能宁可写重，不可写漏。

---

## 0. Frontmatter（YAML 元数据）【必填】

```yaml
---
name: <技能英文名，kebab-case>
description: >
  <一句话定位 + 触发词清单 + 不适用场景>
  触发词包括："xxx"、"xxx"、"xxx"。
  不适用于：xxx（请使用 yyy 技能）、xxx。   # ← 不适用场景必须写，防止误触发
target_role: <目标使用角色，如：信贷风险分析师>
business_domain: <业务域 > 子域 > 场景>
risk_level: high          # 风控技能默认 high；high = 结论必须人工确认
version: 1.0.0
status: draft             # draft → review → published
data_sources:             # 数据源依赖清单（用户上传 / MCP工具 / references）
  - <数据项及来源>
upstream_skills: []       # 前置技能
downstream_skills: []     # 输出消费者
---
```

## 1. 目标角色与使用场景【必填】

- **角色**：谁用
- **使用场景**：什么时点用（贷前尽调 / 审批答辩 / 贷后巡检…）
- **输出用途**：输出给谁、用来干什么
- **决策层级**：⚠️ 必须写明"信息辅助，不得直接作为审批/决策结论"
- **执行频率**：按需 / 定时 / 事件触发

## 2. 数据接入【必填】

### 2.1 必需数据表

| 数据项 | 来源 | 获取方式 | 敏感级别 |
|--------|------|---------|----------|
| 例：财务报表三表 | 用户上传 | 文件上传 | 内部 |
| 例：舆情信息 | tavily 搜索工具 | 工具调用 | 公开 |

### 2.2 数据脱敏规则（输出中如何掩码）

- 统一社会信用代码：前 6 后 4，中间 `*`
- 银行账号：仅显示后 4 位
- 身份证号：前 3 后 4，中间 `*`
- <按需补充本技能涉及的敏感字段>

### 2.3 降级策略（每种数据缺失都要有明确动作）

| 缺失情况 | 动作 |
|----------|------|
| 核心数据为空 | **立即终止分析，不输出任何内容（含标题）** |
| 例：报表仅 1 年 | 标注"数据不足，趋势分析不可用"，仅做静态分析 |
| 例：行业数据不可用 | 标注"未纳入行业对标维度"，仅做个体分析 |

> 原则：降级有据，缺什么标注什么；**严禁用行业均值/估计值顶替真实数据**。

## 3. 术语消歧【必填】

把本技能用到的、容易被模型混淆的术语逐条定义。例：

| 术语 | 本技能中的精确含义 |
|------|--------------------|
| DSCR | 经营现金流 / 当期应还本息（不是 EBITDA/利息） |
| 合并 vs 母公司口径 | 同口径才能对比，严禁混用 |

## 4. 执行流程（Workflow）【必填】

> **先读后写**：任何推理开始前，先列出全部输入数据、确认时间范围 / 口径 / 准则，
> 验证通过才进入步骤 1。

固定步骤框架，每步必须包含三要素：

```
### 步骤 N：<步骤名>
<做什么、查什么、引用哪个 references 文件或工具>

**防偷懒指令：**
- 不得跳过任何维度，即使"看起来正常"
- 必须展示"中间跳板"，不能直接说"A 导致 C"
- 证据不足时承认不足，不捏造

出口路由：
- ✅ <条件> → 进入步骤 N+1
- ⚠️ <条件> → 标注"（数据不足，待核实）"后继续
- ❌ <条件> → 终止，不输出

> 📋 数据来源：`user_upload` / `tool:<工具名>` / `reference`
> 📋 执行主体：`ai` 或 `ai→human`（高风险结论步骤必须 ai→human + approve）
```

最后一步（结论/建议步）**必须** `executor: ai→human`、确认机制 `approve`。

## 5. 输出格式【必填】

- 结论必须**量化**：带具体阈值（如"DSCR < 1.2 时建议降额"），禁止"风险较高"这类空话
- 传导路径用**链式表达**："收入降 X% → 毛利率变化 Y pct → 净利润减 Z% → DSCR = W"
- 每个数据引用必须标来源（报表类型 + 年份 + 科目）
- 不确定性显式标注：基于推断的结论标"（推断）"，缺数据标"（数据不足，待核实）"
- 给出可被下游技能/系统解析的字段规范表

## 6. 合规红线（Constraints）【必填，以下 8 条为风控技能通用底线】

1. **无数据时终止**：核心数据缺失 → 立即终止，不输出任何内容
2. **禁止捏造数据**：只用原始数据中明确出现的数字，引用与原文完全一致（含单位、小数位）
3. **禁止确定性承诺**：不得出现"预计恢复""有望好转""回收率预计 X%"等表述
4. **禁止数据猜测**：缺失按降级策略处理，不得用均值/估计值顶替
5. **口径一致性**：不同口径数据严禁混用对比
6. **数据时效性**：引用超过有效期的基准数据必须标注"⚠️ 可能已过时"
7. **正反证据均衡**：每个风险点同时呈现支持证据与弱化例外，避免单向强化
8. **禁止越权决策**：只输出分析与建议，不得代替人工审批或自动执行调整

（按具体技能补充第 9 条起的专属红线）

## 7. 审计追踪（Audit Trail）【必填】

每次执行生成审计日志 `audit/{对象简称}_{日期}_{技能名}_audit.json`：

```json
{
  "skill_name": "",
  "skill_version": "",
  "execution_time": "ISO8601",
  "operator": "[姓名（工号）]",
  "target": "[脱敏后的分析对象]",
  "steps": [
    { "step": "", "executor": "ai|ai→human", "data_source": {"type": ""}, "result": "通过|降级|预警|终止" }
  ],
  "final_conclusion_confirmed_by_human": false
}
```

## 8. 踩坑记录（Gotchas）【建议】

记录本技能迭代中发现的模型常见错误及对策，持续追加。

## 9. 示例（Examples）【建议】

至少一组完整的输入 → 输出示例；risk_level: high 的技能建议补一组
"数据不足 → 正确终止"的反例。

## 附：点金 credit-risk-manager 的 8 个技能（选题参考）

| 技能 | 定位 | 我们的可移植性 |
|------|------|----------------|
| credit-risk-cot | 给定数据+目标风险点 → 六步证据链 | ★★★ 依赖用户上传财报，改造成本低 |
| loan-risk-monitor | 批量企业司法/资产/经营/信用/舆情五维扫描 | ★★★ 舆情维度可直接用 tavily 搜索 |
| credit-policy-analysis | 信贷政策环境分析 | ★★ 需政策库或搜索 |
| credit-industry-rule-gen | 行业风险规则生成 | ★★ |
| credit-risk-extraction | 历史案例 → 风险分析框架 | ★★ 知识沉淀型 |
| vlm-verifier | 跨模态材料核验 | ★ 需视觉模型配置 |
| post-loan-management | 贷后全周期管理 | ★★ |
| credit-large-exposure-mgmt | 大额风险暴露集中度 | ★★ 需组合数据 |

> 建议起步：先按本模板写一个你自己业务里最高频的风控场景，跑通
> "降级策略 + 出口路由 + 人工确认"三件套，再批量铺开。
